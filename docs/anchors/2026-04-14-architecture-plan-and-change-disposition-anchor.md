# 2026-04-14 架构校准、修正版计划骨架与既有改动处置锚点

## 目的

本文档用于给后续 AI / 开发者提供一个统一锚点，说明：

1. 当前对 MarketRadar 真实模块接线与定位的理解
2. 为什么需要修正此前 roadmap / 推进顺序
3. 修正版计划骨架是什么
4. 已有改动中哪些应保留、哪些应回退、哪些应重构、哪些应清理
5. 后续推进时的边界条件与优先级

这份文档不是最终架构白皮书，而是**当前阶段的决策锚点**。后续若继续演进，应优先基于本文档理解“为什么这样定”，避免重复把实验链能力过早塞进主链。

---

## 一、当前确认的模块接线与定位

### 1. 最小主生产判断链

当前最小主生产判断链应理解为：

- M1 → M2 → M3 → M4

对应含义：

- **M1**：把原始信息翻译为结构化 `MarketSignal`
- **M2**：信号存储与历史检索，承担时间维度记忆层
- **M3**：机会判断层，负责把信号组合成 `OpportunityObject`
- **M4**：行动设计层，负责把机会转成 `ActionPlan`

### 2. 输入前端

- **M0**：信息收集层，是系统与外部世界的输入接口
- M0 不做判断，只负责采集、去重、写入标准输入

### 3. 闭环后段

- **M5**：持仓管理层
- **M6**：复盘归因层

这两者属于系统闭环后段，是完整闭环的重要组成，但不是当前 `pipeline/run_pipeline.py` 的最小前段主链。

### 4. 支撑/证据与执行推进层

- **M8**：知识库/证据基础层，为 M3 / M4 提供历史案例与框架支持
- **M9**：价格更新 / 模拟执行支撑层，为 M5 / 模拟盘 / price loader 服务

### 5. 并行输入层

- **M10**：情绪感知层

当前合理定位：
- 独立采集
- 形成标准情绪信号
- 注入 M2
- 供 M3 未来按标准信号消费

当前不应定位为：
- 直接挂在 M3 主链里的强制后处理增强器

### 6. 验证链 / 高阶实验层

- **M11**：多 Agent 市场情绪模拟 / 叙事模拟层
- **backtest / ablation / strategy backtest**：验证链

当前明确判断：
- M11 已进入工程验证链、实验链、CLI 链
- M11 当前**不是**主生产判断链的必经节点
- M11 当前更适合留在验证链 / 旁路参考层 / 条件门控候选层

### 7. 编排层

- **M7 Scheduler**：系统编排与定时执行层

负责调度：
- signal_pipeline
- news_collect
- sentiment_collect
- price_update
- daily_review

## 二、需要修正此前计划的原因

此前推进中存在若干结构性风险：

### 1. 把验证链能力过早视作主链能力

典型表现：
- 因为 M11 在实验中看起来有效，就倾向于把它升格为主链必经节点
- 因为 M10 有情绪信号，就倾向于直接把情绪共振硬塞进 M3 主链

当前判断：
- **验证链有效 ≠ 应立即进入主链**
- 是否进入主链，必须由稳定验证证据、失败降级路径和清晰角色边界共同决定

### 2. 过度聚焦前段 M1~M4，低估闭环后段的重要性

此前若 roadmap 长期只围绕：
- provider
- M3 增强
- 情绪增强
- agent 增强

则会导致系统变成“很会分析，但闭环不完整”的系统。

当前判断：
- M5 / M6 / M8 / M9 的地位需要上调
- 尤其执行后状态推进、复盘归因、证据沉淀不应长期缺位

### 3. Runtime 问题不是普通 provider 切换问题，而是架构锚点问题

此前部分计划把 runtime 理解为：
- “先找一个更稳定的 provider 跑通”

当前修正为：
- 当前阶段优先目标是**确认主链实际 runtime 到底是什么**
- 即确认真实 provider / model / auth / module route 是否与项目要求一致

因此：
- runtime 修复属于“架构校准前置条件”
- 不是一个普通功能任务

### 4. 需要区分“有没有接进去”和“应不应该成为必经节点”

后续 AI 必须严格区分这两件事：

- 某模块已在工程中可运行 / 已接验证链
- 某模块是否应该进入主生产判断链作为必经路径

这两者不是同一件事。

## 三、修正版计划骨架

### Phase 1：架构校准与链路验真

目标：
- 校准真实主链、支撑层、并行输入层、验证链、编排层边界
- 校准真实 runtime
- 校准既有 roadmap 与当前事实之间的偏差

重点：
- 最小主链：M1 → M2 → M3 → M4
- 输入前端：M0
- 闭环后段：M5 / M6
- 支撑层：M8 / M9
- 并行输入层：M10
- 验证链：M11 / backtest / ablation
- 编排层：M7

### Phase 2：主链稳态建设（M1 → M2 → M3 → M4）

目标：
- 先把主链做稳、做可解释、做可测试

当前优先保留与继续方向：
- M3 parser / JSON 解析稳态
- schema 归一化
- 枚举容错
- 机会对象结构化增强
- ActionPlan 统一语义层

当前不应优先做的：
- 把 M10 / M11 的高阶逻辑直接塞进 M3 主链

### Phase 3：并行输入层建设（重点 M10）

目标：
- 把 M10 做成独立、稳定、可追溯、可存储的情绪感知层
- 通过 M2 注入标准情绪 signal

当前阶段：
- M10 是并行输入层
- 不是 M3 的强制后处理增强器

### Phase 4：验证链建设（backtest / ablation / M11）

目标：
- 让主链是否接入增强能力，由证据决定，而不是由直觉决定

重点：
- historical backtest
- ablation
- strategy backtest
- M11 calibrate / compare / stability analysis

当前明确约束：
- **M11 不应在未充分验证前成为主生产判断链的必经节点**

### Phase 5：闭环后段建设（M5 / M6 / M8 / M9）

目标：
- 补齐执行后状态推进、复盘归因、知识沉淀、价格驱动层

### Phase 6：主链扩容准入机制

任何模块/增强能力想进入主链前，必须回答：

1. 它属于哪一层？
   - 主链基础模块
   - 并行输入层
   - 验证链
   - 旁路参考因子
   - 条件门控候选

2. 它失败会不会阻断主链？
3. 它是否已有稳定验证证据？
4. 它在主链里的角色是什么？
   - 基础事实输入
   - 证据支撑
   - 风险提示
   - 加分器
   - 条件门控
   - 否决器

角色不清楚，不升格。

## 四、关于 M10 与 M11 的明确决策

### 1. M10（情绪层）

当前阶段合理定位：
- 独立采集
- 形成标准情绪 signal
- 注入 M2
- 未来由 M3 先作为普通 signal 消费

当前阶段不应做：
- 继续推动“情绪共振后处理器”直接进入 M3 主链

### 2. M11（多 Agent 模拟）

当前阶段明确结论：
- M11 有价值
- M11 已进入验证链/实验链
- 但 M11 **不应在未经过充分数据验证前成为主生产判断链的必经之路**

原因：
- 输出稳定性仍需验证
- 失败降级路径不够清晰
- 角色边界（加分器/提示器/门控器）尚未完全定型
- 过早升格会把高波动模块直接引入主链

## 五、关于 M4 与策略层的明确决策

### 1. 现状判断

当前仓库中已存在策略原型：
- `backtest/strategies.py`
- `backtest/strategy_backtest.py`

已有基础策略示例：
- MacroMomentum
- PolicyBreakout
- ComboFilter

这说明：
- 项目不是完全没有策略层
- 策略层目前主要存在于 backtest / 验证链

### 2. 是否必须立即把策略上提到 M4？

当前结论：
- **不必立即全量上提到 M4**
- **短期策略继续留在 backtest 层做验证是合理的**

理由：
- 主链当前优先目标是稳态建设
- 不应为架构整齐而强行把主链复杂化

### 3. 但为什么仍建议做“公共策略接口层”？

原因：
- 如果策略永远只留在 backtest，主链与验证链会长期割裂
- 回测验证出的策略结论难以自然反哺 M4 / M5 / M6 / M9

### 4. 当前折中方案（已确认）

当前决策是：

- **短期：策略继续留在 backtest 做验证**
- **近期：抽公共策略接口层**
- **暂不要求：M4 立即多策略输出**

必须明确：
- **共享策略接口层 ≠ 立即多策略主链输出**

这条说明必须写清，避免后续 AI 误以为要马上把 M4 做成多策略总控。

## 六、既有改动的处置结论

以下处置结论是当前阶段的正式建议。

### A. 建议保留

#### 1. Runtime 主链修复
涉及：
- `config/llm_config.yaml`
- `core/llm_client.py`
- `integrations/gongfeng_llm_client.py`
- `run_dev_pipeline.ps1`（若只是配合当前 runtime 主链）
- `docs/LLM_Config.md`

理由：
- 符合“架构校准与链路验真”阶段目标
- 属于基础设施与 runtime 锚点修复

#### 2. pytest 收敛与 M3 健壮性测试
涉及：
- `pytest.ini`
- `tests/test_m3_judgment_engine.py`

理由：
- 符合主链稳态建设
- 支撑 M3 parser / schema / JSON / 枚举归一化稳定性

#### 3. 策略原型本身
涉及：
- `backtest/strategies.py`
- `backtest/strategy_backtest.py`

理由：
- 是 M4 策略模块化的重要前身
- 已有结构价值和验证价值

### B. 建议保留，但重构/收口

#### 4. M4 Action Designer
涉及：
- `m4_action/action_designer.py`

当前判断：
- 保留统一 ActionPlan 语义层
- 后续应逐步支持共享策略接口层
- 但暂不要求 M4 立刻多策略输出

#### 5. 策略测试原型
涉及：
- `tests/_strategy_backtest_test.py`

建议：
- 内容保留
- 正规化命名与结构
- 改造成正式 pytest 测试

#### 6. runtime / parse-failure anchor 机制
建议：
- 保留“正式锚点文档”机制
- 但收敛数量，只保留少数关键样本

### C. 暂缓推进 / 降级优先级

#### 7. M10 → M3 二阶情绪共振扩展
结论：
- 暂缓
- 当前阶段不应继续扩展主链里的情绪共振增强

#### 8. M11 升格为主链必经节点
结论：
- 暂缓
- 当前继续留在验证链 / 旁路参考层 / 条件门控候选层

#### 9. runtime 未收口前继续扩展复杂功能面
结论：
- 统一降级优先级

### D. 建议回退

#### 10. M3 情绪增强主链残留
涉及：
- `m3_judgment/sentiment_resonance.py`
- 以及任何等价的直接挂主链的情绪后处理增强逻辑

结论：
- 回退/删除是对的
- 不建议恢复进主链
- 当前已核：正式 `m3_judgment/` 目录下已无 `sentiment_resonance.py` 文件；残留引用主要存在于历史脚本型测试或注释中，不属于当前正式主链

### E. 建议清理 / 归档

#### 11. 根目录明显临时文件
例如：
- `_tmp_inspect_provider.py`
- `_tmp_scan_llm_refs.py`
- `test_pipeline_run.log`
- `test_pipeline_stdout.log`

结论：
- 清理

#### 12. 待判断是否正规化保留的临时工具
例如：
- `run_smoke_test.ps1`
- `run_test_pipeline_force_xfyun.py`
- `run_test_pipeline_xfyun.ps1`
- `test_auto_fallback.py`

结论：
- 若长期有用，则正规化移入 `scripts/` 或正式测试目录
- 否则清理

#### 13. tests 目录中的历史脚本型测试
例如：
- `_m0_test.py`
- `_m1_test.py`
- `_m3_test.py`
- `_m11_test.py`
- `_sentiment_test.py`
- `_timing.py`
- `_ping.py`
- `smoke_test.py`
- 其它同类下划线测试/脚本

结论：
- 需要一次 tests 专项整治
- 原则：
  - 可重复且有价值 → 正规化保留
  - 手工脚本 → 移出 `tests/`
  - 一次性排障 → 删除

#### 14. 过量的 docs/anchors 调试文档
例如：
- 大量 `overnight-llm-runtime-cycle-*`
- 大量 `m3-stepb-parse-failure-*`

结论：
- 保留少数关键文档与代表样本
- 其余归档或清理

## 七、当前执行顺序建议

- 保留 runtime 主链修复

## 八、给后续 AI 的硬约束

后续 AI 在继续推进前，必须遵守以下约束：

1. **不要把实验链有效性自动等同于主链准入资格**
2. **不要在未验证前恢复 M3 主链中的情绪共振增强**
3. **不要默认推动 M11 成为主链必经节点**
4. **不要把“共享策略接口层”误解为“立即多策略主链输出”**
5. **当前阶段优先级是：runtime 与主链稳态 > 清理噪音 > 小步重构 > 验证链继续扩展**

---

## 九、一句话摘要

当前阶段的总策略是：

> 保留 runtime 主链修复、pytest 收敛、M3 健壮性测试和 backtest 策略原型；回退 M3 情绪共振主链残留；清理临时脚本/日志/过量锚点；M4 保留现有 ActionPlan 语义层，但只做“共享策略接口层”的小步重构，暂不要求主链多策略输出；M11 继续留在验证链，不在未充分验证前升格为主链必经节点。
