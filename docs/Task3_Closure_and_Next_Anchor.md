# Task 3 收口总结与后续迭代锚点

## 1. Task 3 目标回顾

Task 3 的目标是把原先散落在代码里的风险预算、计划有效期、分阶段执行模板、评分卡边界约定等，收敛成一套可维护的参数系统，为后续 M4、回测、模拟盘共用同一语义打基础。

---

## 2. 本轮已完成内容

### 2.1 参数系统文件落地
已新增：
- `config/risk_config.yaml`
- `config/execution_config.yaml`
- `config/opportunity_rules.yaml`

### 2.2 M4 已接入参数系统
当前 `m4_action/action_designer.py` 已消费：
- 优先级风险预算
- 行动计划有效期
- 品类级默认止损/止盈
- 默认 review triggers
- 分阶段建仓模板

### 2.3 模板选择逻辑已明确
M4 当前对分阶段执行模板的选择顺序为：
1. LLM 显式输出的 `phases`
2. `instrument_type` 对应模板
3. `default` 模板

### 2.4 边界已经明确
- 参数系统不推翻 M3 主判断
- `opportunity_score` 仍属于 M3 的解释层输出
- 参数系统当前主要服务于 M4、回测、模拟盘的统一语义

### 2.5 联调验证通过
在 DeepSeek 下，`M1 -> M3 -> M4` 主链路已多次跑通，说明本轮参数外置化未破坏现有流程。

---

## 3. Task 3 当前状态判断

### 已达成
Task 3 的第一阶段目标已经完成：
- 参数已从硬编码迁出
- M4 已具备参数化雏形
- 不同优先级与不同品类已可走不同默认模板

### 未完成但不应在本轮继续深入的事项
以下内容应作为后续优化项，而不是在 Task 3 中无限扩张：
- 参数 schema 与严格校验器
- 参数热加载/动态刷新
- 组合级风险预算
- 多机会之间的风险冲突解决
- 回测/模拟盘全面消费参数系统

也就是说：
**Task 3 现在已经达到了“可以作为后续模块锚点”的完成状态，继续深挖边际收益开始下降。**

## 4. 后续优化方向（保留为迭代锚点）

### 4.1 参数系统自身优化
- 为三份 YAML 增加 schema 校验
- 增加参数默认值检查与启动时告警
- 增加 README / docs 中的参数调整指南

### 4.2 M4 进一步参数化
- 区分 ETF / 股票 / 期货 / 债券 / 指数的更细模板
- 增加按机会类型（政策 / 主题 / 事件驱动）区分模板
- 引入 plan refresh / review cadence 参数

### 4.3 与回测系统打通
- 将 `risk_config.yaml` 直接映射到 backtest risk budget
- 将 `execution_config.yaml` 映射到持有期、止损、止盈、分批建仓模拟
- 使用 `opportunity_rules.yaml` 作为评分卡解释边界约束

### 4.4 与模拟盘系统打通
- 模拟仓位直接读取风险预算与仓位上限
- 模拟持仓生命周期使用统一 valid_until / stop_loss / take_profit 语义
- 将 review trigger 纳入模拟盘检查逻辑

## 5. 下一阶段建议锚点

进入 Task 4 前，建议把下面两点作为固定锚点：

### 锚点 A：参数系统是“共享语义层”，不是“新裁决层”
- 它约束执行与验证
- 但不推翻 M3 的机会判断

### 锚点 B：Task 4 要做的是“参数映射到回测/模拟盘”，不是重复发明一套参数
- 后续所有回测与模拟盘设计，都优先复用 Task 3 已建立的参数系统
- 若发现参数不够，再补参数系统，而不是在回测/模拟盘里私自再造一套

---

## 6. 建议的下一步

Task 4 应优先开展：
1. 设计 `OpportunityObject -> BacktestTask` 的标准映射
2. 设计 `ActionPlan -> 模拟执行参数` 的标准映射
3. 明确回测/模拟盘如何消费 `risk_config.yaml` 与 `execution_config.yaml`

这样推进可以保证系统继续沿着统一语义演进，而不是重新分叉。
